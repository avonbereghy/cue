Why this exists

Claude Code runs in your terminal.
That's great — until it doesn't.

You kick off a long task and switch windows. Five minutes later you glance back: has it finished? Is it waiting for you to approve something? Did it error out silently?

Claude Code is powerful. But it gives you no ambient signal. No visual heartbeat. No quick way to track multiple sessions without keeping a wall of terminals open.

Cue is that ambient signal.

What's inside

Everything you need to know about a session.
Nothing you don't.

Status at a Glance

Five dot states tell the full story: blinking white (working), cyan (subagent), yellow (waiting for approval), red (error), green (done). You know the moment something needs you.

Multi-Session Grid

Track up to 8 Claude Code sessions simultaneously in a compact dot grid. Each dot is a live session — click one to open its dashboard. No more juggling terminal windows.

Token Metrics

Input, output, and cache token counts update in real time. Tokens-per-second shows generation speed. The context bar shows exactly how close you are to the limit.

Permission Controls

When Claude Code pauses to ask permission, Cue surfaces the request with Approve and Deny buttons. You don't need to find the right terminal — it comes to you.

Subagent Awareness

When Claude Code spawns subagents, Cue shows the active count and individual status dots. You can see the whole tree of parallel work happening across agents.

Privacy by Design

Zero outbound network calls. It reads Claude Code's local session files — no data leaves your machine. No analytics, no telemetry, no account required. Fully auditable source.

Common questions

How does Cue know what Claude Code is doing?

Claude Code has a native event system called hooks that fires on every lifecycle event — session start, tool calls, permission requests, completions. Cue registers a lightweight hook that writes session state to a local JSON file on each event.

The desktop app reads that file and parses Claude Code's JSONL conversation logs incrementally (only new bytes each cycle) to extract token counts, tool usage, and context metrics. No API keys needed, no network involved.

Does it work with multiple sessions at once?

Yes. Cue tracks up to 8 concurrent Claude Code sessions simultaneously. Each appears as a dot in the grid. Sessions are detected automatically when Claude Code starts writing to a new session file. You can click any dot to expand the full session dashboard.

macOS says the app is from an unidentified developer

Because Cue is built from source and not notarized with Apple, macOS Gatekeeper will block it on first launch.

Option A (GUI): Right-click the .app in Finder, select Open, then click Open in the dialog. You only need to do this once.

Option B (Terminal): Run xattr -cr /Applications/Cue.app then launch normally. This removes the quarantine attribute without disabling Gatekeeper globally.

Does it send any data anywhere?

No. Cue makes zero outbound network connections. It only reads local files that Claude Code writes to your machine. There is no telemetry, no analytics, no crash reporting, no update check. The source code is fully auditable on GitHub.

Which Claude Code versions and models does it support?

Cue reads the session files Claude Code generates, so it works with whatever version of Claude Code you have installed. The model name displayed (e.g. claude-opus-4-5) is read directly from the session data — no hardcoded model list. Context limits are auto-detected: 1M for Opus/Sonnet 4.6, 200K for older models.

Is a Homebrew formula or binary release planned?

Not yet. Right now the only install path is building from source. I wanted to get the tool out and useful before spending time on distribution infrastructure. If there's demand, a Homebrew cask and pre-built binaries are the natural next step. Open an issue on GitHub if this matters to you — that's the best signal.
