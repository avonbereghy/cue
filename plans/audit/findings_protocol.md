# Protocol Conformance Findings

## Summary
Audited Cue's hook-event and transcript-JSONL assumptions against (a) ~240 live transcripts under `~/.claude/projects/` and (b) the actual Claude Code 2.1.161 Bun-compiled binary (`~/.local/share/claude/versions/2.1.161`), grepping its embedded Zod hook schemas. The core protocol handling is largely correct and well-verified: all 15 installed hook events are real CC events; the Notification `notification_type` field name and its `permission_prompt`/`elicitation_dialog`/`idle_prompt` subtype strings match the binary's matcher exactly; the `PermissionRequest` HTTP response shape (`decision:{behavior:"allow"|"deny"}`) matches the binary's `PermissionRequest` output schema verbatim; `stop_reason` lives in `message`, usage/cache token field names, ISO-8601 timestamps, `isSidechain` filtering, separate `subagents/*.jsonl` routing, AskUserQuestion/ExitPlanMode tool_use->tool_result resolution, and the 1M-model binary scanner all confirm against live data. Two genuine conformance gaps surfaced: the JSONL parser reads only the legacy `custom-title` and ignores the now-dominant `ai-title`/`aiTitle` entry (present in 168 of 169 titled transcripts), and the Python hook's bounded tail-reads frequently fall short of the last assistant line.

## Findings

### F-protocol-001: Parser ignores `ai-title`/`aiTitle` -- the title Claude Code now emits for almost every session
- **Severity:** high
- **Confidence:** 92
- **Files:** cue-desktop/src-tauri/src/jsonl_parser.rs:330-341, 974-977, 1151-1164
- **What:** The parser only recognises `entry_type == "custom-title"` reading the `customTitle` field. But current Claude Code writes the session's title as a separate entry type: `{"type":"ai-title","aiTitle":"...","sessionId":...}`. Live data: `ai-title` appears in 168 transcript files; `custom-title` in 1 (an old retenir log). The binary confirms both code paths exist -- `if(this.currentSessionAiTitle)$.push({type:"ai-title",aiTitle:...})` (auto-generated, written for essentially every session) vs. `{type:"custom-title",customTitle:_.trim()}` (only the explicit user `/title` command and the fork "(Branch)" marker). So `custom-title` is the rare user-set path; `ai-title` is the common auto path the parser never reads.
- **Why it matters:** `SessionMetrics.custom_title` is essentially always `None` on real sessions, so the frontend subtitle/disambiguation hint (`SessionCard.tsx` renders `metrics.customTitle`) is blank for nearly all cards -- the auto-generated human-readable session title never reaches the UI. This is a display-accuracy regression, not a state-machine error; the workspace-leaf fallback still renders, so cards aren't broken, just less informative and ambiguous when two sessions share a workspace.
- **Suggested fix:** In `parse_line`, add a branch that reads `aiTitle` from `type=="ai-title"` entries into `entry.custom_title` (run the same `strip_ansi/strip_xml/strip_bracket` sanitisation). Keep the existing `custom-title` branch so explicit `/title` and the `(Branch)` fork detection at lines 1151-1164 (which depends on the `(Branch)` suffix that only `custom-title` carries) still work -- i.e. an explicit `custom-title` should win over `ai-title` when both exist.
- **Verification:** `grep -l '"type":"ai-title"' ~/.claude/projects/*/*.jsonl | wc -l` -> 168 vs `grep -l '"type":"custom-title"'` -> 1. Binary: `grep -a -o '.\{40\}type:"ai-title".\{0,60\}' <claude-bin>` shows the auto-write call site. Sample entry: `{"type":"ai-title","aiTitle":"Review client house purchase scenarios","sessionId":...}`.

### F-protocol-002: Hook tail-reads miss the last assistant line on a large fraction of transcripts
- **Severity:** high
- **Confidence:** 80
- **Files:** hooks/cue-hook:496-518 (`_read_last_assistant`, 16 KiB), 555-594 (`_read_last_tokens`, 8 KiB)
- **What:** Both helpers `seek(-chunk, 2)` and read only the last `min(size, 8192|16384)` bytes, then `splitlines()` and scan backward for the last `type=="assistant"` line. Real transcripts routinely push the last assistant entry outside that window: measured over 116 transcripts containing >=1 assistant line, the last assistant line starts before the 8 KiB window in 47.4% of files and before the 16 KiB window in 5.2%. Drivers: individual JSONL lines reach 2.2 MB (max observed); 7.5% of all lines exceed 8 KiB; and trailing entries are frequently non-assistant rows (`system`, `file-history-snapshot`, `last-prompt`, `ai-title`, and `user` tool_result lines up to 20 KB) that consume the window. When the assistant line is missed, `_read_last_tokens` returns `{}` and `_read_last_assistant` returns `None`.
- **Why it matters:** Three hook behaviors degrade. (1) Token/model write-through (`inputTokens`/`outputTokens`/`model` in sessions.json) silently falls back to the stale prior value or empty -- the dashboard's token/context pill lags until the 5 s Rust full-file re-parse catches up (Rust reads the whole file, so it self-corrects -- this is the mitigation that keeps it HIGH not CRITICAL). (2) `_last_tool_was_ask_question` (done/idle->waiting fast-path) can miss an AskUserQuestion at the tail and emit `idle`/`done` instead of seeding `waiting`; the Rust `awaiting_user_prompt` verdict still promotes within ~1 s, so the user sees a brief wrong state, not a stuck one. (3) `_turn_has_finished` (SubagentStop routing, line 898) can wrongly return `False` and route a finished turn to `working` instead of `idle` -- again corrected by the Rust turn-ended pass. Net: transient wrong states + lagging token display, bounded by Rust authority.
- **Suggested fix:** Raise the tail-read chunk (e.g. 256 KiB) and/or loop: if no `assistant` line is found in the first window, double the chunk and re-read up to a cap (matching the 1 MiB stdin cap already used elsewhere). Alternatively, since `parse_jsonl_to_session_metrics` on the Rust side is the authority for all of these, demote the hook's reads to "best-effort hint" and document that they may be empty (the code already tolerates `{}`/`None`, so correctness holds -- this finding is about reducing the transient-wrong-state and token-lag window).
- **Verification:** Reproduce with: for each transcript, compute the byte offset of the last `type=="assistant"` line and compare to `filesize-8192` / `filesize-16384`. Script output over 116 files: 55 (47.4%) miss 8 KiB, 6 (5.2%) miss 16 KiB. grep-measured line sizes: max 2,204,685 bytes; 4,464/59,713 lines (7.5%) >8 KiB.

## Out of scope

Verified-correct (no finding -- recorded so the next auditor needn't re-derive):
- All 15 HOOK_EVENTS are real (env_detect.rs:169-185): each `hookEventName` literal -- including the less-obvious `PostToolUseFailure`, `StopFailure`, `TaskCompleted`, `SubagentStart`/`Stop`, `PreCompact`/`PostCompact`, `PermissionRequest` -- is present in the 2.1.161 binary's Zod schemas.
- Notification subtype dispatch is correct (hooks/cue-hook:671-682): the binary's notification matcher is `{fieldToMatch:"notification_type",values:["permission_prompt","idle_prompt","auth_success",...]}` and `case"Notification":$=K.notification_type`. The field name (`notification_type`, not `type`/`subtype`) and the `elicitation_dialog`/`permission_prompt`/`idle_prompt` strings all match the hook exactly.
- PermissionRequest response shape is correct (permission_server.rs:88-91): binary schema is `hookEventName:literal("PermissionRequest"),decision:union([{behavior:literal("allow"),updatedInput?...},{behavior:literal("deny"),message?,interrupt?}])`. Cue's `ALLOW_RESPONSE`/`DENY_RESPONSE` match. (Note: the *PreToolUse* hook uses a different `permissionDecision:"allow"|"deny"|"ask"` shape, but Cue correctly targets `PermissionRequest`, not PreToolUse, for its allow/deny forwarding.) The hook->server forwarded fields (`session_id`, `tool_name`, `tool_input`, `hook_event_name`) and the server's snake/camel-tolerant parse (lib.rs:1700-1724) align.
- stop_reason / usage / model / timestamp field shapes (jsonl_parser.rs:418-499, 615-652): live assistant line has `message.stop_reason` (top-level absent), `message.usage.{input_tokens,output_tokens,cache_creation_input_tokens,cache_read_input_tokens}`, `message.model`, and ISO-8601 string `timestamp` ("2026-06-03T00:25:36.767Z") -> parser's Format-2 path. Distinct `stop_reason` values seen: `tool_use` (68k), `end_turn` (4.8k), `stop_sequence`, `max_tokens`, null -- parser handles `tool_use`/`end_turn` and ignores the rest, which is correct. The nested `usage.cache_creation` object is correctly ignored in favor of the flat `*_input_tokens`.
- Subagent handling (jsonl_parser.rs:843-937, 1166-1186): 83 `subagents/` dirs on disk; entries carry `agentId` + `isSidechain:true`; companion `.meta.json` has `{agentType,description,toolUseId}` (parser reads `description`). `slug` is absent on current entries (None) but the parser tolerates that. No `isSidechain:true` lines appear in any main transcript (0/all files) -- confirms current CC routes subagents to separate files and the main-transcript `isSidechain` filter (line 312) is a correct forward-compat no-op.
- awaiting_user_prompt (jsonl_parser.rs:1102-1118): live AskUserQuestion/ExitPlanMode tool_use carries `id`; the answer is a `type:"user"` entry with `content[].{type:"tool_result",tool_use_id}` matching -- exactly the resolution the verdict implements. 174 AskUserQuestion + 21 ExitPlanMode tool_uses observed.
- Subagent/Todo tool names (jsonl_parser.rs:469-489, 512-533): the spawn tool is named `Agent` in live data (903 uses), and `extract_tool_target` already special-cases `"Agent"`. `TaskCreate`/`TaskUpdate` dominate (1296/2414) over the near-extinct `TodoWrite` (6); `TaskCreate` input is `{subject,description,activeForm}` (parser reads `subject` first) and `TaskUpdate.taskId` is a numeric string `"1"`,`"3"` matching `Task #N` creation order, so the parser's 1-based index logic is correct.
- model_context.rs binary scanner: the live binary contains `permission_prompt`(17)/`hookSpecificOutput`(31)/etc., confirming it is the Bun-compiled bundle the scanner targets; the 1M-model gate-function approach and `FALLBACK_1M_SUBSTRINGS` (opus-4-6/7/8, sonnet-4-0/5/6) are consistent with the live `claude-opus-4-8` model string seen in transcripts. (Did not run the `#[ignore]` live_binary test, but the scanner architecture matches the observed binary shape.)
- session_type / print-mode ignore (hooks/cue-hook:426-439): binary contains `session_type`, `"interactive"`, `"print"` -- `_should_ignore_session`'s `session_type != "interactive"` test is on the right field/values.
- Dead `entry_type == "tool_result"` branch (jsonl_parser.rs:504-506): confirmed 0 top-level `type=="tool_result"` entries across all live transcripts (tool_results are nested in `type:"user"` content blocks, which the parser handles at 396-411). The branch is harmless dead code and the code comments already acknowledge it -- not a finding.

Could NOT fully verify:
- Notification(permission_prompt) -> pending_tool_use timing: confirmed `permission_prompt` is a real subtype and that a permission-gated tool leaves an unresolved tool_use at the tail, but did not capture a live permission-prompt Notification payload mid-flight (would require an interactive session blocking on a prompt during the audit). The seed->resolve logic is internally consistent and the field names check out; the runtime timing (does the hook fire before/after the unresolved tool_use is flushed to JSONL) was not directly observed.
- StopFailure `error_type` value set (hooks/cue-hook:698-707): `StopFailure` is a real event and `case"StopFailure"` exists in the binary, but I did not enumerate the exact `error_type` string set from the binary to confirm the hook's documented list (`rate_limit`/`billing_error`/...) is complete and current. The hook passes `error_type` through as an opaque string, so an unknown value degrades gracefully -- low risk regardless.
