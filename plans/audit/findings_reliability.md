# Reliability Findings

## Summary
The poll loop and JSONL parser are well-hardened against malformed/truncated input and won't panic on bad transcript data. The two material reliability gaps are (1) the Python hook's main-path locked write has no outer exception guard — a lock acquisition failure either silently drops the state update (Unix, after the 2s flock timeout) or hangs the hook until Claude Code kills it (Windows, where msvcrt.locking blocks with no timeout), and (2) the Rust backend uses poison-on-panic std::sync::Mutex with .lock().unwrap() in ~60 places plus a non-aborting panic hook, so a single panic while any shared lock is held permanently wedges the dashboard until restart.

## Findings

### F-reliability-001: Hook main-path locked write silently drops the update (Unix) or hangs the hook (Windows) when `_lock()` raises
- **Severity:** high
- **Confidence:** 85
- **Files:** hooks/cue-hook:747-749, hooks/cue-hook:30-44, hooks/cue-hook:18-26, hooks/cue-hook:1000-1005
- **What:** In `main()`, the read-modify-write critical section is wrapped in `with open(LOCK_FILE, "a+b") as lock:` and `_lock(lock)` is called BEFORE the `try:`/`finally:` block (lines 747-750). There is no outer `try/except` around `main()`'s locked write (contrast `_quick_state_write` at lines 248/327-328, which wraps everything in `try: ... except Exception: pass`). On Unix, `_lock` (lines 30-41) retries non-blocking `flock` for 2s then `raise`s the `IOError/OSError`; that exception propagates out of `main()` uncaught, so the state write never lands and Claude Code sees a stderr traceback. On Windows, `_lock` (lines 18-22) calls `msvcrt.locking(..., LK_LOCK, 1)` which BLOCKS indefinitely (no timeout, unlike the Unix path) — under contention the hook stalls until Claude Code's 5s hook timeout kills it, again dropping the update.
- **Why it matters:** Under concurrent hook activity (multiple sessions firing events, or a held lock) a real transition — working→idle, the amber waiting flip, error — is silently lost, so the dashboard lies about session state until the next successful hook event. On Windows the indefinite block is worse: it consumes the whole 5s budget per contended event.
- **Suggested fix:** Wrap the main-path locked write in the same `try/except Exception: pass` (or at minimum around `_lock`) that `_quick_state_write` already uses, so a lock failure degrades to a no-op instead of an uncaught traceback. Separately, give the Windows `_lock` a bounded retry using `LK_NBLCK` in a timed loop (mirroring the Unix 2s deadline) instead of the unbounded blocking `LK_LOCK`.
- **Verification:** `grep -n "_lock(lock)" hooks/cue-hook` shows the call sits outside any try in `main` (~line 749) and inside a try in `_quick_state_write` (~line 251); `grep -n "LK_LOCK\|LK_NBLCK" hooks/cue-hook` confirms the Windows path uses the unbounded blocking variant.

### F-reliability-002: Poison-on-panic `std::sync::Mutex` + non-aborting panic hook can permanently wedge the dashboard
- **Severity:** high
- **Confidence:** 70
- **Files:** cue-desktop/src-tauri/src/session_monitor.rs:50-106, cue-desktop/src-tauri/src/session_monitor.rs:163, cue-desktop/src-tauri/src/lib.rs:1264-1272, cue-desktop/src-tauri/src/lib.rs:257
- **What:** All shared state is held in `std::sync::Mutex` (session_monitor.rs:50-106) and accessed almost exclusively via `.lock().unwrap()` (41 in session_monitor.rs, 19 in lib.rs). `std::sync::Mutex` poisons itself if a thread panics while holding the guard; afterward every `.lock().unwrap()` on that mutex panics too. The custom panic hook (lib.rs:1264-1272) only logs and returns — it does NOT abort (no `panic="abort"`, no `process::exit`). So if any code inside a held-lock scope panics — an unexpected arithmetic/index panic in a `poll_status` transform, or the spawn_blocking closures that clone `enriched_sessions.lock().unwrap()` — that mutex stays poisoned for the process lifetime. Every later `poll_status`, `refresh_metrics`, the 250ms blink tick (lib.rs:2288), and the `get_sessions` command (lib.rs:257) that touches it then panics on its own `.unwrap()`, freezing the tray and dashboard with no self-recovery.
- **Why it matters:** A long-running daemon must never wedge. One transient panic in any lock-holding path becomes a permanent process-wide freeze: the UI stops updating and never recovers until relaunch. The non-aborting hook makes this worse than a crash (a crash with autostart would restart clean).
- **Suggested fix:** Either (a) switch shared state to `parking_lot::Mutex` (no poisoning), or (b) replace `.lock().unwrap()` with `.lock().unwrap_or_else(|e| e.into_inner())` to recover through poison, or (c) make the panic hook abort so a poisoned lock can't linger. (a)/(b) preserve availability; (c) trades it for a clean restart.
- **Verification:** `grep -c "lock().unwrap()" cue-desktop/src-tauri/src/session_monitor.rs cue-desktop/src-tauri/src/lib.rs` (60 total); `grep -rn "parking_lot\|panic *=" cue-desktop/src-tauri/Cargo.toml` confirms std Mutex and no abort strategy; inspect lib.rs:1264-1272.

## Out of scope

- **JSONL parser robustness (jsonl_parser.rs):** Thoroughly hardened — `parse_line` is `?`/`.ok()`-based, `cap_snippet` respects char boundaries, `refresh_entry_cache` only consumes up to the last `\n`, truncation/mtime-regression/deletion/oversize reset the cache, `O_NOFOLLOW` guards symlinks. Hostile-input tests (lines 1969-2065) close the panic surface.
- **`poll_status` async correctness:** Correctly inside `tokio::task::spawn_blocking` from the 1s/5s loops, so long reads + the first-call model_context scan don't block the runtime; `interval` skips missed ticks and awaits the prior task, so slow polls delay rather than overlap; locks dropped before the `enriched_sessions` write.
- **Blink timer (250ms) + theme poll (2s) run sync work on the async runtime (lib.rs:2283-2299, 1402-1440):** Blink loop calls `settings::load_settings()` + tiny-skia rendering directly on the async worker; small/OS-cached I/O, lock held only for a clone. Minor politeness issue, not a wedge. settings.rs/tray.rs out of scope.
- **`permission-log.jsonl` unbounded growth (permission_log.rs:24-50):** Append path has no rotation/size cap; read path is bounded at 16 MiB so it can't OOM. Disk-leak only, read-safe — below the HIGH floor.
- **Permission server resource safety (lib.rs:1500-1832, permission_server.rs):** Well-bounded — MAX_PENDING=64, 60s timeout frees pending+metadata, header/body caps (8 KiB/1 MiB), Host + constant-time token auth, metadata-before-pending insert order.
- **sessions.json self-repair / read resilience (session_monitor.rs:158-261):** Parse- and read-failure arms preserve prior state for 5 polls before destructive action; FileTooLarge keeps prior state. Sound.
- **model_context.rs binary scanning:** Bounded windows + string-aware brace matching (caps 4096/512/80), slice-safe `rfind`/`find_all`/`literals_after`, 512 MiB read cap, OnceLock single-run. No panic/OOB surface.
- **paths.rs `home_dir().expect(...)` (paths.rs:95):** Panics if home unresolvable, but runs at startup and is unrecoverable anyway. Acceptable fail-fast.
